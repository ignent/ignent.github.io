Linux 服务器部署

提示

Linux 不是 “怎么一键优化”，而是 “先理解系统，再决定要不要动它”。

Linux 初始化配置

新机器拿到手，第一步通常不是先配业务，而是先把系统更新到最新状态。

sudo apt update
sudo apt full-upgrade -y
sudo apt autoremove -y

如果时区不对，很多日志和定时任务都会看着很别扭，先改掉：

sudo timedatectl set-timezone Asia/Shanghai
timedatectl status

服务端通用必备软件

够用就好，不用为了凑全家桶装太多。

sudo apt install -y \
  sudo curl wget git vim nano unzip zip tar rsync tmux htop vnstat lrzsz \
  lsof jq tree ca-certificates gnupg bash-completion

如果经常需要排查网络、域名或端口，也可以把这些诊断工具一并装上：

sudo apt install -y \
  dnsutils net-tools iproute2 traceroute mtr-tiny

常用工具

htop

htop 是我最常开的进程查看器，比 top 顺手很多，适合看 CPU、内存、线程、进程树和 load。

Debian / Ubuntu 直接装：

sudo apt install -y htop
htop

常用操作通常记住这几个就够了：

/    搜索进程
\    过滤进程
t    切换树状视图
.    切换排序列
k    杀进程

htop 更适合在界面里配，不太建议手搓配置文件。直接按 F2 进 Setup，把 CPU、Memory、Load Average、IO 或者进程字段调成自己习惯的样子即可。

它默认会把用户配置写到：

~/.config/htop/htoprc

这个文件会在 htop 正常退出时被重写，所以真要改配置，优先还是在界面里改。

vnstat

vnstat 很适合拿来看服务器网卡的日流量、月流量和计费周期，比临时抓包轻得多。

安装并启动：

sudo apt install -y vnstat
sudo systemctl enable --now vnstat

第一次安装完，先确认你的网卡名，再初始化数据库：

ip -br link
sudo vnstat -u -i eth0
vnstat -i eth0

常用查看方式：

vnstat -d -i eth0
vnstat -m -i eth0
vnstat -l -i eth0

这里分别是看日统计、月统计和实时流量。 vnstat 的数据库默认放在 /var/lib/vnstat/ ，配置文件默认是：

/etc/vnstat.conf

我更常改的是这几项：

HourlyDays
DailyDays
MonthlyMonths
MonthRotate
MaxBWeth0

MonthRotate 适合按账单周期算月流量，不一定非得每月 1 号切。 MaxBWeth0 适合给某块网卡设一个上限，避免异常采样把统计写脏。改完以后重启服务：

sudo systemctl restart vnstat

lrzsz

lrzsz 是老牌的 ZMODEM/XMODEM/YMODEM 文件传输工具。现在很多人直接用 scp 、 rsync ，但如果你在 Xshell 、 SecureCRT 、 MobaXterm 这类支持 ZMODEM 的终端里，它依然很好用。

安装：

sudo apt install -y lrzsz

最常用的其实就两个命令：

rz
sz filename

rz 是把本地文件传到服务器， sz 是把服务器文件传回本地。多文件发送也可以：

sz file1 file2

这套东西没有太多要配的，关键是你的 SSH 客户端得支持 ZMODEM。浏览器里的 Web SSH、很多简化终端、以及某些 tmux / screen 场景里，自动弹窗传输不一定能正常工作，这种时候直接换回 scp 或 rsync 更省心。

lrzsz 默认还是比较保守的：接收端默认是受限模式，不接受绝对路径或父目录引用，也不会直接覆盖已有文件。真要深挖参数，直接看：

man rz
man sz

dnslookup

这里说的 dnslookup 不是系统自带的 nslookup ，而是一个支持普通 DNS、DoT、DoH、DoQ、DNSCrypt 的命令行查询工具，排查现代 DNS 解析链路时会比传统工具顺手。

如果机器上已经有 Go，最简单是直接装：

go install github.com/ameshkov/dnslookup@latest
sudo install -m 0755 ~/go/bin/dnslookup /usr/local/bin/dnslookup

如果你不想在服务器上装 Go，也可以直接从它的 GitHub Releases 下载对应架构的预编译二进制。

常见用法：

dnslookup example.org
dnslookup example.org 1.1.1.1
dnslookup example.org tls://dns.adguard.com
dnslookup example.org https://dns.google/dns-query

查特定记录类型，或者直接拿机器可读输出：

RRTYPE=AAAA dnslookup example.org 1.1.1.1
JSON=1 dnslookup example.org 1.1.1.1
DNSSEC=1 dnslookup example.org tls://8.8.8.8

它基本不是靠配置文件工作的，而是更偏一次性命令行参数和环境变量。常见的几个开关就是：

RRTYPE
JSON
DNSSEC
VERIFY
HTTP3

比如测试自签名 DoT/DoH 环境时，临时关证书校验可以这样：

VERIFY=0 dnslookup example.org tls://127.0.0.1

但这种写法更适合测试，不适合当成长期默认配置。

NextTrace

NextTrace 适合拿来做路由追踪、回程测试和链路观察，特别是国内多运营商、多区域场景下，比传统 traceroute 更直观。

Debian / Ubuntu 上我更建议直接用它官方提供的 APT 仓库：

sudo install -d -m 0755 /etc/apt/keyrings
curl -fsSL -o /tmp/nexttrace-archive-keyring.gpg https://github.com/nxtrace/nexttrace-debs/releases/latest/download/nexttrace-archive-keyring.gpg
sudo install -m 0644 /tmp/nexttrace-archive-keyring.gpg /etc/apt/keyrings/nexttrace.gpg
rm -f /tmp/nexttrace-archive-keyring.gpg
printf '%s\n' 'Types: deb' 'URIs: https://github.com/nxtrace/nexttrace-debs/releases/latest/download/' 'Suites: ./' 'Signed-By: /etc/apt/keyrings/nexttrace.gpg' | sudo tee /etc/apt/sources.list.d/nexttrace.sources >/dev/null
sudo apt update
sudo apt install -y nexttrace

如果你只想要轻量版，也可以装：

sudo apt install -y nexttrace-tiny

常见命令：

nexttrace 1.1.1.1
nexttrace --tcp --port 443 example.com
nexttrace --udp 1.0.0.1
nexttrace --fast-trace
nexttrace -t 1.1.1.1
nexttrace --dev eth0 2606:4700:4700::1111
nexttrace --mtu 1.1.1.1

这几个分别对应普通追踪、TCP/UDP 探测、一键回程测试、MTR 模式、绑网卡和 PMTU 探测。

NextTrace 不是必须先配文件才能用，但它支持从下面的位置发现配置：

$XDG_CONFIG_HOME/nexttrace/nt_config.yaml

另外有些参数更适合放环境变量，比如：

export NEXTTRACE_MAXATTEMPTS=10
export NEXTTRACE_IPINFO_TOKEN=your-token

要注意一点：探测节奏这类参数目前更偏命令行使用，不是所有运行参数都会存进 nt_config.yaml 。如果你有自己常用的探测组合，很多时候写成 shell alias 反而更实用。

chsrc

chsrc 是一个换源工具，适合一台机器上同时处理系统源、语言包管理器和常见软件源。前面那段 Debian 换源是手写配置的做法，而 chsrc 更像是把这件事工具化。

Linux 上最省事的是官方脚本安装：

curl https://chsrc.run/posix | bash

非 root 默认会装到 ~/.local/bin ，root 安装则默认进 /usr/local/bin 。如果你更想走 deb 包，也可以下载它官方提供的 .deb 再用 apt install ./xxx.deb 安装。

上手先看它支持什么：

chsrc ls
chsrc ls debian
chsrc ls pip

看当前配置、测速、自动换源：

chsrc get pip
chsrc measure pip
chsrc set pip
sudo chsrc set debian

如果你不想现场测速，而是直接用维护团队给出的第一推荐源：

chsrc set pip first
sudo chsrc set debian first

回滚到默认上游：

chsrc reset pip
sudo chsrc reset debian

chsrc 比较值得记住的是这几个选项：

-dry
-scope=project|user|system
-ipv6

-dry 可以先看它准备改什么再决定是否执行； -scope 很适合只改当前项目或当前用户，不一定非要全局动系统；系统级目标像 debian 、 ubuntu 这类通常还是要 sudo 。

Java 服务端部署

Java 机器建议把 “运行时” 和 “构建环境” 分开看。只跑现成 jar 的机器装运行时就够了，要在服务器上编译或排障，再补 JDK 和构建工具。

只需要运行 Java 程序：

sudo apt install -y default-jre-headless

如果还要在服务器上构建、排查线程或内存问题：

sudo apt install -y default-jdk maven

有些 Java 服务会生成 PDF、图片或报表，缺字体时容易出怪问题，可以顺手补一个：

sudo apt install -y fontconfig

Python 服务端部署

Python 机器的关键不是 “全局 pip 装一堆”，而是先把 venv 和编译依赖准备好，再把项目依赖装进虚拟环境。

sudo apt install -y \
  python3 python3-venv python3-pip python3-dev \
  build-essential pkg-config

如果项目会连 PostgreSQL 或 MySQL，这些头文件包经常会用到：

sudo apt install -y libpq-dev default-libmysqlclient-dev

初始化一个常见的 Python 项目环境：

python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip

如果要安装 MySQL

在 Debian 上，最省事的装法通常不是直接找 Oracle MySQL 包，而是安装 default-mysql-server 。要先知道一点：这条路大概率装下来的是 MariaDB，而不是 Oracle MySQL。

sudo apt install -y default-mysql-server

装完以后先确认服务起来了。Debian 上服务名通常是 mariadb ：

sudo systemctl enable --now mariadb
sudo systemctl status mariadb

第一次安装完，建议先跑一遍安全初始化：

sudo mariadb-secure-installation

有些环境里也可能是这个命令：

sudo mysql_secure_installation

登录数据库，通常可以先用系统 root 直接进：

sudo mysql

初始化一个业务库和业务账号，别让应用直接拿 root 跑：

CREATE DATABASE appdb CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'appuser'@'127.0.0.1' IDENTIFIED BY 'change-this-password';
GRANT ALL PRIVILEGES ON appdb.* TO 'appuser'@'127.0.0.1';
FLUSH PRIVILEGES;

这里我更倾向于先给 127.0.0.1 ，意思是只允许本机应用通过 TCP 连，不要一上来就开放成 '%' 。

如果要看当前账号和库，大致用这几条就够了：

SHOW DATABASES;
SELECT user, host FROM mysql.user;

MySQL 安装时注意的几件事

1. 先分清你要的是 MySQL，还是 MySQL 协议兼容数据库

如果项目只是说 “用 MySQL”，大多数时候 MariaDB 就够了；但如果项目明确依赖 MySQL 8 的某些行为、函数、驱动兼容性或运维规范，就不要默认走 default-mysql-server ，而应该直接按官方文档安装 Oracle MySQL，并且不要和系统默认 MariaDB 混装。

2. 不要直接把 3306 暴露到公网

很多时候数据库只该给本机应用或者内网机器访问。上线前至少看一眼监听地址和端口：

sudo ss -lntp | grep 3306

如果只是单机应用，本地回环或者内网访问通常就够了，真正要对外开放时再配防火墙白名单。

3. 业务账号不要偷懒用 root

应用应该用单独账号、单独库、最小权限。这样后面换密码、做审计、做隔离都轻松很多。

4. 字符集直接用 utf8mb4

新项目没必要再从 utf8 、 latin1 这些历史包袱开始。建库、建表时直接统一到 utf8mb4 ，省得后面为 emoji、特殊字符和排序规则补坑。

5. 备份要先于上线

哪怕只是个人项目，也别等出事以后才想到导数据。最朴素的逻辑备份命令就够先用起来：

mysqldump --single-transaction --routines --triggers appdb > appdb.sql

如果后面数据量起来了，再考虑二进制日志、物理备份和定时备份策略。

如果是前端项目

前端不能只看成 “纯静态”，至少要分成三种。

1. 纯静态托管

如果前端已经在别处构建好，这台机器只负责托管 dist 、 build 之类的静态产物，通常只需要 Nginx：

sudo apt install -y nginx

这种场景一般不需要在生产机上安装 Node.js。

2. SSR / Node 常驻型前端

如果跑的是 Next.js、Nuxt、Remix 这类需要 Node 常驻进程的前端服务，这台机器本质上已经是一个 Node 服务端。最少要有 Node.js、npm，再配一个反向代理。

sudo apt install -y nodejs npm nginx
node -v
npm -v

如果项目用的是 pnpm ，可以补上：

sudo npm install -g pnpm
pnpm -v

常见流程一般是：

npm install
npm run build
npm run start

这种机器除了装 Node.js，本地还要考虑进程守护。更稳妥的做法通常是写 systemd 服务，让前端进程跟着系统启动和重启。

3. 只负责构建的前端机器

还有一种机器不直接对外提供页面，只负责拉代码、装依赖、打包前端产物。这种机器不一定需要 Nginx，但一定需要 Node.js 和包管理器。

sudo apt install -y nodejs npm git

如果构建过程依赖原生模块，也可以顺手补上编译环境：

sudo apt install -y build-essential pkg-config python3

这样像 sharp 、 node-gyp 这类依赖出问题时，排查会轻松很多。

Debian 换源要跟上 DEB 822

这段配置是给 Debian 13 写系统源，位置放在 /etc/apt/sources.list.d/debian.sources 。

Types: deb
URIs: https://mirrors.sustech.edu.cn/debian/
Suites: trixie trixie-updates
Components: main contrib non-free non-free-firmware
Signed-By: /usr/share/keyrings/debian-archive-keyring.gpg

Types: deb
URIs: https://mirrors.sustech.edu.cn/debian-security/
Suites: trixie-security
Components: main contrib non-free non-free-firmware
Signed-By: /usr/share/keyrings/debian-archive-keyring.gpg

旧格式还能用：

deb https://mirrors.sustech.edu.cn/debian/ trixie main contrib non-free non-free-firmware
deb https://mirrors.sustech.edu.cn/debian/ trixie-updates main contrib non-free non-free-firmware
deb https://mirrors.sustech.edu.cn/debian-security trixie-security main contrib non-free non-free-firmware

写完以后刷新索引：

sudo apt update

如果机器上还留着旧格式源，可以顺手转换：

apt modernize-sources

先建普通用户，再给 sudo

这两条命令是最基础的最小权限实践，别长期拿 root 当日用账号。

sudo adduser $USERNAME
sudo adduser $USERNAME sudo

等价写法：

sudo useradd -m -G sudo -s /bin/bash $USERNAME
sudo passwd $USERNAME

SSH 先配密钥，再收紧入口

先生成密钥对：

ssh-keygen

再把公钥写入 authorized_keys ：

cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys

真正关键的是服务端配置。不要直接改 /etc/ssh/sshd_config ，而是新建覆盖文件：

sudo tee /etc/ssh/sshd_config.d/00-override.conf >/dev/null << 'EOF'
Port $PORT
PermitRootLogin no
PasswordAuthentication no
PermitEmptyPasswords no
PrintLastLog yes
ClientAliveInterval 30
ClientAliveCountMax 3
EOF

这段配置的用途：改高位端口、禁 root、禁密码登录，再用 ClientAliveInterval 和 ClientAliveCountMax 给 SSH 会话做保活。

配完 SSH 之后，先测配置再重载：

sudo sshd -t
sudo systemctl reload sshd

新开一个窗口试连：

ssh -i /path/to/your/private/key $USERNAME@$IP

如果不想每次都敲长命令，可以在本地 ~/.ssh/config 里写一份别名：

Host $NICKNAME
  HostName $IP
  User $USERNAME
  Port $PORT
  IdentityFile /path/to/your/private/key
  ServerAliveInterval $SECONDS
  ServerAliveCountMax $TIMES

# 第一台服务器：公司开发机（使用密钥登录）
Host dev
    HostName 192.168.1.50
    User ubuntu
    IdentityFile ~/.ssh/id_rsa_company

# 第二台服务器：个人博客（使用自定义端口和密码）
Host blog
    HostName example.com
    User root
    Port 2222

# 第三台服务器：阿里云国际站（使用另一个密钥）
Host ali-us
    HostName 47.88.1.1
    User admin
    IdentityFile ~/.ssh/id_ed25519

之后直接连：

ssh $NICKNAME

UFW

sudo apt install ufw
sudo ufw status

启用前先放行 SSH 端口，修改端口的配置文件在 /etc/ssh/sshd_config ：

sudo ufw allow $PORT/tcp

检查规则：

sudo ufw status

确认无误后再启用：

sudo ufw enable

sudo ufw allow ssh ，通常只会放 22 端口，不一定是真正改过的 SSH 端口。

Fail2ban

如果 SSH 已经改成纯密钥登录，继续给 SSH 上 Fail2ban 收益很有限；但如果还有通过用户名 + 密码认证的服务，它依然是有价值的。

BBR

Debian 13 以后， sysctl 配置更适合写到 /etc/sysctl.d/ 。BBR 开启方式是：

echo "net.ipv4.tcp_congestion_control=bbr" | sudo tee /etc/sysctl.d/99-bbr.conf
sudo sysctl --system

.bashrc 可以做一点顺手优化

alias ll='ls -l'
export PATH=$PATH:/sbin

前者是常用别名，后者是把 /sbin 补进普通用户的 PATH ，免得一些系统命令明明能用却找不到。
