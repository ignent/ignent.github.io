Nginx 常用配置

ZIP2026/5/21...大约 5 分钟

提示

这篇文档默认以 Debian 系服务器为例，重点讲日常最常见的 Nginx 运维方式，配置该怎么理解、怎么改、怎么上线。

一、适用场景

这篇文档适合这些场景：

  • 纯静态站点托管
  • SPA 前端站点
  • 反向代理 Java / Python / Node 服务
  • HTTPS、跳转、上传、日志这类日常运维配置

二、安装与目录结构

先安装 Nginx：

sudo apt update
sudo apt install -y nginx
nginx -v

Debian 系最常见的目录结构大致是：

/etc/nginx/
├── nginx.conf
├── conf.d/
├── sites-available/
└── sites-enabled/

最常见的分工是：

  • nginx.conf ：全局主配置
  • sites-available/ ：每个站点的独立配置文件
  • sites-enabled/ ：启用中的站点，一般是软链接

三、常用部署操作

Nginx 日常运维最常用的命令就这几组：

测试配置是否有语法问题：

sudo nginx -t

重新加载配置，不中断已有连接：

sudo systemctl reload nginx

重启服务：

sudo systemctl restart nginx

看运行状态：

sudo systemctl status nginx

四、站点配置的标准落地方式

比较稳妥的做法是每个站点一个文件，先写到 sites-available ，确认无误后再链接到 sites-enabled 。

比如创建一个站点：

sudo tee /etc/nginx/sites-available/demo >/dev/null <<'EOF'
server {
    listen 80;
    server_name demo.example.com;
    root /srv/www/demo/current;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}
EOF

启用这个站点：

sudo ln -sfn /etc/nginx/sites-available/demo /etc/nginx/sites-enabled/demo
sudo nginx -t
sudo systemctl reload nginx

如果要停用：

sudo rm -f /etc/nginx/sites-enabled/demo
sudo nginx -t
sudo systemctl reload nginx

五、 server 块说明

绝大多数日常站点配置，核心都在 server { ... } 里。你可以把它理解成 “一个站点 / 一个域名 / 一类流量” 的处理规则。

先看一个典型例子：

server {
    listen 80;
    server_name demo.example.com www.demo.example.com;
    root /srv/www/demo/current;
    index index.html index.htm;
    client_max_body_size 20m;
    access_log /var/log/nginx/demo.access.log;
    error_log /var/log/nginx/demo.error.log warn;

    location / {
        try_files $uri $uri/ /index.html;
    }
}

下面这些属性最常用。

1. listen

表示这个站点监听哪个端口。

listen 80;
listen 443 ssl http2;

最常见的是：

  • 80 ：HTTP
  • 443 ：HTTPS

2. server_name

表示这个 server 块匹配哪些域名。

server_name demo.example.com www.demo.example.com;

如果请求的域名匹配上了，Nginx 才会走这个站点配置。

3. root

表示站点根目录，适合静态文件站点。

root /srv/www/demo/current;

通常和 index 、 try_files 一起出现。

4. index

表示默认首页文件名。

index index.html index.htm;

5. client_max_body_size

表示允许上传的最大请求体大小。

client_max_body_size 20m;

如果你有文件上传，没配这一项时很容易遇到 413 Request Entity Too Large 。

6. access_log

表示访问日志文件路径。

access_log /var/log/nginx/demo.access.log;

7. error_log

表示错误日志文件路径和日志级别。

error_log /var/log/nginx/demo.error.log warn;

8. location

这是 server 里最关键的子块，用来决定不同路径怎么处理。

比如：

location / {
    try_files $uri $uri/ /index.html;
}

它的意思是：先找真实文件，再找目录，最后回退到 index.html 。

9. return

通常用来做跳转或直接返回状态码。

return 301 https://$host$request_uri;

常见用法是把 HTTP 强制跳到 HTTPS。

10. rewrite

用来重写 URL，但它比 return 更容易让配置变复杂。简单跳转优先用 return ，不要一上来就写复杂 rewrite 。

六、 location 常见写法

1. 纯静态站点

location / {
    try_files $uri $uri/ =404;
}

适合普通 HTML、图片、CSS、JS 文件站点。

2. SPA 单页应用

location / {
    try_files $uri $uri/ /index.html;
}

适合 Vue、React、Angular 这类前端路由站点，刷新页面不会 404。

3. 反向代理后端服务

location / {
    proxy_pass http://127.0.0.1:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

这组 proxy_set_header 基本可以当默认模板。

4. 独立静态资源目录

location /static/ {
    alias /opt/apps/demo/shared/static/;
}

这里要特别注意： alias 和 root 不一样。

  • root ：把请求路径拼到根目录后面
  • alias ：直接把当前路径段映射到指定目录

5. 上传目录

location /media/ {
    alias /opt/apps/demo/shared/media/;
}

七、常见站点模板

1. 纯静态站点

server {
    listen 80;
    server_name site.example.com;
    root /srv/www/site/current;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}

2. SPA 站点

server {
    listen 80;
    server_name spa.example.com;
    root /srv/www/spa/current;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}

3. Java / Python / Node 反向代理

server {
    listen 80;
    server_name app.example.com;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

4. HTTP 跳 HTTPS

server {
    listen 80;
    server_name app.example.com;
    return 301 https://$host$request_uri;
}

5. HTTPS 站点

server {
    listen 443 ssl http2;
    server_name app.example.com;

    ssl_certificate /etc/letsencrypt/live/app.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/app.example.com/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

八、常用参数说明

除了 server 块本身，日常最常改的参数还有这些。

1. proxy_read_timeout

后端响应较慢时，可以适当调大：

proxy_read_timeout 60s;

2. proxy_connect_timeout

表示连接后端服务的超时时间：

proxy_connect_timeout 10s;

3. proxy_send_timeout

表示把请求发给后端的超时时间：

proxy_send_timeout 60s;

4. gzip

压缩文本类响应，常见配置：

gzip on;
gzip_types text/plain text/css application/json application/javascript application/xml+rss;

5. add_header

常用来加响应头：

add_header X-Frame-Options SAMEORIGIN;
add_header X-Content-Type-Options nosniff;

九、HTTPS 部署

如果域名已经解析到服务器，可以直接用 certbot 自动签发：

sudo apt install -y certbot python3-certbot-nginx
sudo certbot --nginx -d app.example.com

自动续期测试：

sudo certbot renew --dry-run

十、日志与排查

查看 Nginx 状态：

sudo systemctl status nginx

看访问日志：

sudo tail -f /var/log/nginx/access.log

看错误日志：

sudo tail -f /var/log/nginx/error.log

如果某个站点单独定义了日志路径，就直接看那个文件。

十一、常见问题

1. nginx: configuration file test failed

先看报错行号，再回头检查是否少分号、少括号、路径写错：

sudo nginx -t

2. 403 Forbidden

常见原因有三类：

  • 目录权限不对
  • root 或 alias 路径写错
  • 没有可访问的首页文件

3. 404 Not Found

优先检查：

  • 文件是不是真的存在
  • root / alias 是否写对
  • SPA 是否忘了 try_files $uri $uri/ /index.html

4. 502 Bad Gateway

通常不是 Nginx 自己坏了，而是后端服务没起来或者端口不通：

sudo systemctl status demo
sudo ss -lntp | grep 8080

5. 413 Request Entity Too Large

说明上传大小超限，补上：

client_max_body_size 20m;

十二、上线检查清单

  • 站点配置写在 sites-available/
  • sites-enabled/ 已正确软链接
  • nginx -t 通过
  • systemctl reload nginx 成功
  • 域名、端口、证书、日志路径都已核对
  • 静态站点、SPA、反向代理这三类场景没有写混
  • server 块里的 listen 、 server_name 、 root 、 location 、 proxy_pass 都已逐项确认
