Web 部署

ZIP2026/5/20...大约 5 分钟

提示

这里把前端项目分成三类来看：纯静态托管、SSR / Node 常驻型前端、只负责构建的前端机器。不要把它们混成一种运维方式。

一、适用场景

这篇文档适合这些场景：

  • 纯静态站点托管
  • Next.js/ Nuxt / Remix 这类 SSR 前端
  • 前端构建机、打包机

二、机器初始化

先把系统和基础工具装好：

sudo apt update
sudo apt full-upgrade -y
sudo apt autoremove -y
sudo apt install -y \
  curl wget git vim unzip zip tar rsync tmux htop lsof jq \
  ca-certificates gnupg nginx ufw

如果机器要跑 SSR 服务，再准备一个服务账号：

sudo adduser --system --group --home /opt/apps/web svc-web
sudo mkdir -p /opt/apps/web/releases /opt/apps/web/shared /data/logs/web
sudo chown -R svc-web:svc-web /opt/apps/web /data/logs/web

三、环境安装

1. 纯静态托管

纯静态站点只要有 Nginx 就够了：

nginx -v

2. SSR / Node 常驻型前端

如果要跑 Next.js、Nuxt 这类服务，还要安装 Node.js：

sudo apt install -y nodejs npm
node -v
npm -v

如果项目用的是 pnpm ，可以补上：

sudo npm install -g pnpm
pnpm -v

3. 只负责构建的前端机器

构建机场景通常还会用到这些依赖：

sudo apt install -y build-essential pkg-config python3

四、项目目录规划

静态站点更适合放到 /srv/www ：

/srv/www/site
├── current -> /srv/www/site/releases/2026-05-20-120000
└── releases
    ├── 2026-05-19-100000
    └── 2026-05-20-120000

SSR 前端更像常规应用，建议放到 /opt/apps/web ：

/opt/apps/web
├── current -> /opt/apps/web/releases/2026-05-20-120000
├── releases
│   ├── 2026-05-19-100000
│   └── 2026-05-20-120000
└── shared
    └── .env

五、部署流程

1. 纯静态托管

先在本地或 CI 把前端打包好，再把产物上传到服务器：

npm install
npm run build

服务器上准备新版本目录：

sudo mkdir -p /srv/www/site/releases/2026-05-20-120000
sudo rsync -av dist/ /srv/www/site/releases/2026-05-20-120000/
sudo ln -sfn /srv/www/site/releases/2026-05-20-120000 /srv/www/site/current

2. SSR / Node 常驻型前端

如果是 SSR 项目，服务器上常见流程是：

sudo mkdir -p /opt/apps/web/releases/2026-05-20-120000
sudo rsync -av ./ /opt/apps/web/releases/2026-05-20-120000/
sudo chown -R svc-web:svc-web /opt/apps/web/releases/2026-05-20-120000

进入新版本目录安装依赖并构建：

cd /opt/apps/web/releases/2026-05-20-120000
sudo -u svc-web npm install
sudo -u svc-web npm run build

如果项目使用 pnpm ：

cd /opt/apps/web/releases/2026-05-20-120000
sudo -u svc-web pnpm install --frozen-lockfile
sudo -u svc-web pnpm build

再准备环境变量：

sudo tee /opt/apps/web/shared/.env >/dev/null <<'EOF'
NODE_ENV=production
PORT=3000
EOF
sudo chown svc-web:svc-web /opt/apps/web/shared/.env

最后切换软链：

sudo ln -sfn /opt/apps/web/releases/2026-05-20-120000 /opt/apps/web/current
sudo chown -h svc-web:svc-web /opt/apps/web/current

六、服务守护与开机自启

1. 纯静态托管

纯静态站点没有应用进程，不需要 systemd 托管业务，只要保证 Nginx 正常即可。

2. SSR / Node 常驻型前端

SSR 项目建议交给 systemd ：

sudo tee /etc/systemd/system/web.service >/dev/null <<'EOF'
[Unit]
Description=demo web service
After=network.target

[Service]
Type=simple
User=svc-web
Group=svc-web
WorkingDirectory=/opt/apps/web/current
EnvironmentFile=-/opt/apps/web/shared/.env
ExecStart=/usr/bin/npm run start
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF

启用并启动：

sudo systemctl daemon-reload
sudo systemctl enable --now web
sudo systemctl status web

如果项目要求自定义端口，也可以把启动命令写成显式形式。

七、网络访问与对外暴露

1. 纯静态站点

静态站点的 Nginx 配置通常这样写：

server {
    listen 80;
    server_name site.example.com;
    root /srv/www/site/current;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}

2. SSR / Node 常驻型前端

SSR 项目通常把 Node 服务绑到 127.0.0.1:3000 ，再由 Nginx 反代：

server {
    listen 80;
    server_name web.example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

写入配置并启用：

sudo tee /etc/nginx/sites-available/web >/dev/null <<'EOF'
server {
    listen 80;
    server_name web.example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
EOF

sudo ln -sfn /etc/nginx/sites-available/web /etc/nginx/sites-enabled/web
sudo nginx -t
sudo systemctl reload nginx

如果要启 HTTPS：

sudo certbot --nginx -d site.example.com -d web.example.com

八、日志与监控

静态站点主要看 Nginx ：

sudo tail -f /var/log/nginx/access.log
sudo tail -f /var/log/nginx/error.log

SSR 项目还要一起看 Node 进程：

journalctl -u web -f
sudo systemctl status web
sudo ss -lntp | grep 3000
curl http://127.0.0.1:3000/

构建时如果卡住，先看这几个：

node -v
npm -v
df -h
free -h

九、备份与回滚

静态站点和 SSR 项目都适合用软链回滚：

sudo ln -sfn /srv/www/site/releases/2026-05-19-100000 /srv/www/site/current

或：

sudo ln -sfn /opt/apps/web/releases/2026-05-19-100000 /opt/apps/web/current
sudo systemctl restart web

需要重点备份的通常是：

  • SSR 项目的 .env
  • 用户上传资源
  • 构建产物
  • Nginx 配置

十、安全加固

  • 纯静态站点不需要装一整套 Node 运行时
  • SSR 项目不要把 Node 进程直接暴露到公网
  • 只开放 22 、 80 、 443
  • 生产机尽量少装前端调试工具

常用防火墙放行：

sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable

十一、常见故障排查

1. 静态资源 404

先看 root 路径和构建产物是否真的上传到了目标目录：

ls -lah /srv/www/site/current
sudo nginx -t

2. 502 Bad Gateway

大多数时候是 SSR 进程没起来：

sudo systemctl status web
journalctl -u web -n 100
sudo nginx -t

3. npm run build 失败

常见原因是 Node 版本不对、磁盘不够、内存不够：

node -v
df -h
free -h

4. 路由刷新 404

如果是 SPA，通常要补 try_files ：

try_files $uri $uri/ /index.html;

5. SSR 启动正常但页面空白

优先看环境变量、接口地址、静态资源前缀和 basePath 配置。

十二、上线检查清单

  • 先分清是静态站点还是 SSR 项目
  • 静态站点已验证 dist 或 build 产物完整
  • SSR 项目已验证 npm run build 和 npm run start
  • systemctl status web 正常
  • nginx -t 通过
  • 域名访问正常，HTTPS 正常
  • 保留旧版本目录，可随时切回
